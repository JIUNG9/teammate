# /attest-compliance

Convert a compliance score run into an auditable artifact: a one-page PDF
with the score, probe results, target commit, and timestamp. Opt-in
sigstore keyless signing produces a `.sig` and `.crt` that any auditor
can verify with `sigstore verify-blob` — pure Python, no `cosign` binary
required.

## When to invoke

- The user asks for "an audit artifact", "a compliance attestation", "a
  signed PDF I can hand to legal".
- After a `score` run that the team wants to preserve as evidence.
- Before a SOC2 / ISMS audit window — auditors love verifiable artifacts.

## Behavior

1. Re-runs the score (so the PDF reflects the current commit, not a stale
   vault entry). Catalogs and probes are loaded from disk.

2. Renders the PDF via `reportlab`:

   - Header: framework names, "compliance attestation"
   - Score block: overall %, count breakdown, target path, commit SHA, captured timestamp
   - Probe table: probe id, result, framework:control mapping, severity
   - Footer: teammate version + verification command snippet

3. If `--sign` is set, mints a sigstore keyless signature via GitHub OIDC.
   Opens an interactive browser flow for the user to authenticate. Splits
   the bundle into `.sig` and `.crt` files. Logs the signing event to the
   public Rekor transparency log.

4. Writes everything to `compliance-vault/attestations/<timestamp>.pdf`
   (and `.sig`, `.crt` if signed) plus a companion `<timestamp>.md`
   describing the artifact in human-readable form.

## Run

```bash
teammate attest                # unsigned preview PDF in the vault
teammate attest --sign         # interactive OIDC sign + .sig + .crt
```

## Output

```
wrote attestation: /path/to/compliance-vault/attestations/2026-04-26-1030.pdf
```

## Verification (for auditors)

The README documents the standard `sigstore verify-blob` command:

```bash
sigstore verify-blob examples/attestation.pdf \
  --signature examples/attestation.pdf.sig \
  --certificate examples/attestation.pdf.crt \
  --certificate-identity https://github.com/placen-org/teammate/.github/workflows/sign-example.yml@refs/heads/main \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
```

If the verify exits 0, the auditor knows:

  - The PDF was generated by teammate at the recorded timestamp
  - The PDF hasn't been tampered with since
  - The cert was issued to a specific GitHub Actions workflow on a
    specific branch — not just "anyone with sigstore"

## What signing does NOT prove

The signed PDF proves *what teammate observed* on the recorded commit.
It does NOT prove the codebase is actually compliant — the score is
mechanical and a team can game it. See `docs/SECURITY.md` for the threat
model.
